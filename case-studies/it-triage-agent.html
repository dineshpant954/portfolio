// 01

Hook & Problem Statement

Every enterprise runs on tickets. Password resets, VPN issues, software requests, access changes — 10,000 to 100,000+ per month flowing through service desks that were designed for a pre-AI world. The economics are brutal: Tier-1 resolution costs ~$22/ticket, but the moment something escalates to Tier-2 or Tier-3, costs jump to $55–$104+ per ticket. And yet, 40–60% of this volume is repetitive, pattern-matchable, and automatable — if you have an agent that can read, classify, reason, and act safely.

This isn't a chatbot that suggests KB articles. This is an agentic system that triages, retrieves knowledge with citations, executes safe resolutions autonomously, and escalates with full context when it can't. The core PM question: when should the agent act, and when should it hand off to a human?

// Success Criteria

Reduce ticket resolution cost by 30%+, cut MTTR from 4.2 hours to under 2 hours, achieve 30%+ deflection rate for Tier-1 volume — while maintaining or improving CSAT and passing security review.

Market Reality & Maturity Path

// What Enterprises Are Trying to Achieve

Every major ITSM vendor is shipping AI capabilities: ServiceNow Now Assist embeds generative AI into workflows, Atlassian Rovo/JSM offers AI-powered virtual agents, Zendesk AI automates ticket routing and suggestions. Microsoft Copilot for Service, Google Vertex AI Agents, and Amazon Bedrock Agents are all entering the space. The market is moving from "should we use AI?" to "how fast can we deploy it safely?"

// What Blocks Progression
  • KB quality debt: 40%+ of knowledge base articles are stale, duplicated, or incomplete. RAG is only as good as its retrieval corpus.
  • Tool-execution risk: Auto-resolving a password reset is very different from auto-modifying Active Directory groups. Risk varies by 100x across intents.
  • Security & compliance gaps: GRC teams need to audit every AI action. Most systems lack the observability infrastructure for this.
  • Organizational adoption: L1 agents fear displacement. L2/L3 SMEs don't trust AI triage. Change management is as hard as the technology.

// 04

Solution Overview (One-Pager)

// What It Does
  • Intake & Classify: Accepts tickets from any channel (Slack, Email, Portal, Teams) and normalizes into a unified schema
  • Smart Triage: Classifies intent, priority, and confidence using Claude API structured output in a single call
  • Knowledge-Grounded Resolution: RAG retrieval from KB + past tickets with citations — never hallucinates a fix
  • Tool Execution: Executes safe resolutions via APIs (Azure AD, Intune, Jamf) with rollback support and audit logging
  • Agent Copilot: For non-auto-resolvable intents, drafts responses and proposes actions for human review
  • Continuous Learning: Feedback loops update KB, retrain classifiers, and adjust confidence thresholds over time
// What It Does NOT Do (Non-Goals)
  • Replace L2/L3 specialists — complex issues always route to humans with full context
  • Auto-resolve security incidents — security intents are hardcoded to escalate, always
  • Modify production infrastructure — no access to prod systems, only IT support tools
  • Store or log PII — all PII masked before LLM processing, no PII in audit trails
  • Work without human oversight — even L3 "lights-out" has policy gates and audit trails
// Value Propositions
For End Users
Get your password reset in 47 seconds instead of 45 minutes. No more waiting in queue, no more repeating your issue to three different agents. The AI resolves routine issues instantly and routes complex ones with full context so the human who picks it up already knows your situation.
For IT Leadership
Reduce cost per ticket from $22 to $15. Cut MTTR by 57%. Deflect 30%+ of volume automatically. Every AI action is audited and policy-gated — you get the efficiency of automation with the governance your security team demands. ROI visible in Week 4.
For L1/L2 Agents
Stop doing password resets all day. The AI handles the routine so you can focus on the interesting, complex issues that actually need your expertise. When tickets do reach you, they arrive with full diagnostic context — no more detective work to figure out what the user actually needs.

Workflow Design (Before/After)

// 1. Password Reset
Before — 15–45 min manual

Manual Process

  • User submits ticket via email/portal
  • Ticket sits in queue (5–20 min)
  • L1 agent picks up, verifies identity manually
  • Agent navigates to Azure AD, resets password
  • Agent emails user the temp password
  • Agent closes ticket manually
After — 47 seconds lights-out

Agent-Resolved (L3)

  • User messages Slack bot
  • Triage agent: account_unlock (97% confidence)
  • verify_identity → Okta Verify push (2s)
  • unlock_account → Azure AD API (1s)
  • reset_password → generate temp (1s)
  • notify_user → Slack DM with link (1s)

Handoff rule: Confidence <90% or identity verification fails → escalate to L1

Audit: TOOL_CALL azure_ad.unlock_account | approver: AUTO (L3 policy) | id: AUD-9821

// 2. VPN Connectivity
Before — 2–6 hours email exchanges

Manual Process

  • User emails IT: "VPN keeps disconnecting"
  • L1 agent asks for VPN client version (email reply wait)
  • User responds hours later
  • L1 searches KB, finds nothing current
  • Escalates to L2 with minimal context
  • L2 re-diagnoses from scratch
After — ~15 min with HITL

Agent-Assisted (L2 HITL)

  • Triage agent: vpn_connectivity (93% confidence)
  • diagnose → Client 6.1.2 matches known issue (3s)
  • enrich → CMDB lookup device info (2s)
  • propose → Push GP 6.1.4 via Jamf → awaiting L1 approval
  • draft_reply → Generated response for agent review

Handoff rule: Software deployment always requires human approval (L2 policy)

Audit: TOOL_CALL jamf.deploy_package | awaiting: L1 approval

// 3. Software Install
Before — 1–3 business days

Manual Process

  • User submits portal request for Figma
  • L1 checks if software is approved (manual lookup)
  • L1 checks license availability (emails vendor admin)
  • Requests manager approval via email
  • Manager approves 1–2 days later
  • L1 submits deployment request to endpoint team
After — ~30 min with approval flow

Agent-Assisted (L2 HITL)

  • Triage agent: software_install (95% confidence)
  • policy_check → Figma approved, Product entitled (1s)
  • license_check → 8/50 available (2s)
  • approval_check → Manager workflow triggered
  • deploy → Queue Intune deployment upon approval

Handoff rule: Manager approval required for all software installs (policy)

Audit: TOOL_CALL intune.deploy_app | approval: pending mgr confirmation

Security, Privacy & Compliance

// Threat Model
ThreatRiskControls
Prompt InjectionHigh — attacker crafts ticket to manipulate agent behaviorInput sanitization, structured output parsing (not free-text), intent classification before action, adversarial test suite
Data ExfiltrationMedium — LLM leaks PII from tickets or KBPII detection + masking before LLM processing, no PII in prompts or logs, data retention policy
Unsafe Tool ExecutionHigh — agent executes destructive actionIntent whitelist, per-tool risk levels, HITL approval for high-risk tools, rollback support
Privilege EscalationHigh — agent gains access beyond scopeLeast-privilege service accounts, scope-limited tokens, no credential storage in agent code
Hallucinated ResolutionMedium — agent invents a fix not grounded in KBRAG with citations (must cite source), confidence thresholds, human review for novel resolutions
Denial of ServiceLow — flood system with tickets to overwhelm agentRate limiting, queue depth monitoring, circuit breaker on agent processing

// Evaluation Harness Plan
  • Golden Set: 200 manually labeled tickets across all intent categories — used for regression testing every release
  • Adversarial Tests: 50 prompt injection attempts, ambiguous tickets, multi-intent tickets, PII-heavy tickets — must pass 100%
  • Regression Suite: Automated nightly runs against golden set — any accuracy drop >2% blocks deployment
  • A/B Testing: Shadow mode for new capabilities — AI processes ticket in parallel, human resolves, compare outcomes before enabling

Roadmap & Execution Plan

// MVP Wedge Rationale

Password reset / account unlock is the ideal MVP wedge: highest volume category (25–35% of tickets), lowest risk (identity verification + single API call), fastest time-to-value (lights-out resolution in under 60 seconds), and most measurable ROI (direct cost comparison to manual process). If we can't get this right, we can't get anything right.

// Interview Talking Points
  • Trust Ramp Tradeoff: "The hardest PM decision isn't what the AI can do — it's what you allow it to do. We designed a 4-level trust ramp because enterprise IT won't accept big-bang autonomy. Each level has to earn its way to the next through measured performance."
  • Security-First Design: "I made security a design principle, not an afterthought. Security incidents are hardcoded to escalate — no confidence score overrides that. Every tool call is audited. PII is masked before it reaches the LLM. This is how you get GRC approval."
  • MVP Wedge Selection: "Password reset was the obvious wedge: highest volume (25–35%), lowest risk (single API call), fastest time-to-value (47 seconds vs. 45 minutes). If we can't nail this, we can't nail anything. Start narrow, prove value, then expand."
  • Build vs. Buy Nuance: "The answer depends on your lens. A startup builds the orchestration and adapters (that's the moat). A consulting firm maximizes reusable patterns on client infra. An enterprise extends their existing ITSM vendor's AI with custom agent logic for gaps."
  • Measuring AI That Acts: "Traditional chatbot metrics (CSAT, containment) don't capture agentic value. You need tool-call correctness, hallucination rate, and citation accuracy alongside deflection rate and MTTR. We designed an evaluation harness with golden sets, adversarial tests, and shadow-mode A/B testing."
